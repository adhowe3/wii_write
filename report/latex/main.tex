\subsection*{Introduction}
% Motivate and introduce the problem you are solving. Very briefly summarize your methodology, your experiments, and your important results.
The issue we are addressing is communication from one user to another via the Wii remote. This communication system could be used by people who lack the fine motor skills to type, or even the ability to talk. This project also serves as an interesting way to learn about Bluetooth and machine learning (ML).

We implement a Bluetooth connection between a Raspberry-Pi to a Wiimote. We design an ML model to detect letters drawn in the air which achieved 97\% accuracy on user dependent data. We experiment on the accuracy of the model and the Bluetooth packet loss at several distances. We find that the model is accurate enough to detect letters correctly and the Wiimote is capable of sending data up to 80ft without significant packet loss. Our code can be found at: \texttt{https://github.com/adhowe3/wii\_write}.

\subsection*{Related Work}
% Summarize the existing work related to your problem.
In previous work, the Wiimote has been reverse engineered and used in virtual reality~\cite{leeHacking} , physical activity tracking~\cite{Khan2013WiiActivity}, and even used to evaluate Parkinson's patients~\cite{SynnottWiiPD}. These works focused on using the Wiimote as an cheap alternative to other motion tracking devices.

There is a significant amount of research about air-writing letter detection. In~\cite{ChenAirWriting} they achieved a word error rate of 0.8\% with air-writing. The work in~\cite{YanayAirwriting} achieved 89.2\% accuracy on a letter prediction model that was user dependent, meaning the model was trained and tested on data from the same user.


\subsection*{Methodology}
% Describe your solution and any other methods you have used to solve your problem.
The Wii remote is connected to the Raspberry Pi over Bluetooth through BlueZ, a native Linux Bluetooth stack. This connection manages the legacy Bluetooth signal from the Wii remote and opens a Logical Link Control and Adaptation Protocol (L2CAP) transport pipe. 
% Our python script for reading the accelerometer data from the Wiimote could interface directly to this stream using a raw Bluetooth socket, but that would bog down the Raspberry Pi's CPU. The Pi would receive 100 accelerometer readings per second, and these would need to be manually decoded from the hexadecimal Wiimote payloads. Instead, to optimize the stream of data and allow enough resources for the ML model on to run the Pi, 
We abstract the handling of the Bluetooth packets to the Linux kernel with the hid-wiimote kernel driver. This translates the Nintendo-specific hexadecimal payloads into standard Linux inputs. Once the payloads are in a format that Linux understands, the evdev (event device) subsystem filters the stream of Bluetooth data and only registers unique events that out python script uses to collect that data for processing. 

We used a Raspberry Pi Zero2W to connect to the Wiimote over Bluetooth. The Rasberry Pi receives raw accelerometer and button data from the Wiimote, and from this data predicts the drawn letter. To draw the letters, we hold the "A" button on the Wiimote and draw a stroke of the letter. Some letters like "H" have three strokes, while a letter like "S" only has one stroke. All leters we draw are upper-case. Accelerometer data is only sent while the "A" button is being held. Once the user is done writing a letter, the D-pad is pressed to indicate the end of letter.

We collected 25 samples of air-writing for every letter. For training, we used 80\% of this data, and the other 20\% was used for testing.

We tried several model architectures, and decided on a hybrid model that takes votes from two individual models. 

The first model is a random forest (RF) classifier, which takes extracted features from the accelerometer data to make a prediction. These features are: number of strokes, length of data, mean, standard deviation, minimum, maximum, for X, Y and Z data. The RF model uses 100 decision trees to predict its output.

The second model is a multi-layer perceptron (MLP) that takes in the raw X, Y, Z accelerometer data and passes it through four fully connected hidden layers with sizes 256 $\rightarrow$ 128 $\rightarrow$ 64 $\rightarrow$ 32. Each layer uses ReLU activation and is trained using Adam optimizer with a learning rate of $1 \times 10^{-4}$. Since the MLP model input needs to have a fixed size, we up-sample or down-sample to 100 samples for each axis input. 

From these two model votes, we created a hybrid model. We trained it to an acceptable accuracy, and implemented a full flow that allowed live data to be acquired, inference executed, and a letter predicted. We tested the hybrid model's accuracy on live data, by air-writing each letter 10 times and evaluating against the model predictions.

Beyond building, training and testing a custom ML model, we also tested the range of the Wiimote with the Raspberry Pi setup. We tested the Wiimote at several distances in three related experiments. These tests are discussed at the bottom of the next section.

In this project, we also experimented with two methods of communicating back to the Wiimote. The first method was using the Wiimote's built in speaker. Because the Wiimote uses a very unique audio discriptor, sending intelligible audio clips was difficult. In the second method we used the vibration and LED on the Wiimote to denote a dot or a dash for morse code communication.


\subsection*{Implementation/Experimentation}
% Describe your experiments and your results. Discuss your results.
Individually, the two models were each about 90\% accurate, but when combine they reached 97\% on the test data. This shows the power of synergy and hybrid models.

The accuracy test on real world live data was 93.08\% accurate. In~\cref{fig:confusion_matrix} we show each letters true label and the model's prediction on the X axis from this live data test. We note here that the letter with the worst performance is "A". Oddly enough, in this test it was confused with "C" the most. In other tests not documented, we saw "A" be confused more with "H" and "I", which make more sense given that they all have three straight strokes.

To evaluate the maximum effective range of the system, we tested packet reception and prediction accuracy at varying distances. 

First, we tested the reliability of a discrete button press signal using a custom script to count detected events. We pressed the 'A' button 50 times at distances ranging from 1 to 90 feet, at 10 foot increments. zero button presses were dropped up to 80 feet away; at 90 feet, 15 out of 50 button presses were lost.

Second, we evaluated continuous accelerometer telemetry to compare the received packet rate against the Wiimote’s 100 Hz transmission rate. Because the evdev subsystem filters non-unique accelerometer data, the remote was continuously shaken to ensure a constant stream of unique spatial events. We recorded the number of packets received during the duration of an 'A' button hold. Due to the Time-Division Multiplexing (TDM) collision on the Raspberry Pi’s shared Wi-Fi and Bluetooth antenna, our maximum receivable baseline was $\sim$75 Hz. Testing at the same distance intervals, we maintained this $\sim$75 Hz baseline up to 70 feet. At 80 feet, throughput fell to $\sim$50 Hz, and degraded to as low as 5 Hz at 90 feet. 

Third, we tested the accuracy of the ML model while performing live inference at several distances. Using the letter 'L' (which achieved 100\% accuracy in our 2-foot baseline tests), we performed 5 gesture writes at each distance. The model successfully maintained 100\% prediction accuracy up to 80 feet, before sharply declining to 20\% accuracy at 90 feet.  

\subsection*{Conclusion}
% Summarize your work and your results. Indicated any future directions.
In this project, we engineered an accurate machine learning pipeline to translate physical gestures into air-written text using a Nintendo Wii remote and a Raspberry Pi Zero 2W. By abstracting the Bluetooth payload to the Linux evdev subsystem of the hid-wiimote driver, we were able to get data to the Raspberry Pi while still managing compute resources for the machine learning model. Our hybrid RF and MLP model was able to get a 93.08\% accuracy rate on live data and our spatial testing proved the system's high resilience, demonstrating that the ML architecture can successfully interpolate telemetry and maintain 100\% prediction accuracy at distances up to 80 feet.

The current state of the project presents a fun proof of concept, but we would like to expand capabilities across hardware, software and security. At the hardware level, we want to eliminate the Time-Division Multiplexing (TDM) bottleneck caused by the Pi's shared antenna by adding a dedicated antenna to receive and transmit Bluetooth. This will also allow the system to run without relying on a WiFi SSH session. 
Regarding security and bidirectional communication, future work will involve developing a lightweight, application-layer encryption handshake. Securing this line of communication will allow us to safely reintroduce bidirectional haptic and audio output to the Wii remote. Future work would also include getting the audio to work on the Wii remote speakers. 
Lastly, we want to improve the machine learning models so that the letter inference is not user dependent. This will likely involve collecting further training data. We also want to implement Natural Language Processing (NLP) autocomplete and spellcheck layer above the character prediction model, allowing the system to contextually correct isolated algorithmic misclassifications (such as the confusion between "A" and "C") in real time.
